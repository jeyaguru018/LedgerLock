PROJECT 2
1. Project Title
FairQueue — Starvation-Free Priority Job Scheduler

2. The Problem
Every backend system eventually needs to run work in the background — sending emails, processing orders, generating reports, retrying failed payments. The moment you have more than one priority level of job, a hidden danger appears: if high-priority jobs keep arriving, low-priority jobs can get pushed back indefinitely and never run at all — a classic problem called starvation. On top of that, jobs fail — networks drop, third-party APIs time out — and naive systems either give up immediately or retry so aggressively they overwhelm the very system they're calling. Most student and even many production systems handle only half of this problem, not both.

3. Existing Solutions
Simple task queues (basic FIFO job queues in many apps).
Enterprise schedulers (Quartz Scheduler, Spring Batch).
Message-queue-based systems (Celery with RabbitMQ, AWS SQS with dead-letter queues).
4. How They Win and How They Lose
Where they win: Simple FIFO queues are easy to build and understand. Quartz/Spring Batch are mature, widely used in enterprise Java. Celery/SQS scale well and are cloud-native.

Where they lose: FIFO queues have no concept of priority at all. Quartz and Spring Batch support priority scheduling but don't guarantee starvation prevention — a flood of high-priority jobs can still permanently starve low-priority ones. Celery/SQS handle retries with backoff but treat all jobs equally within a queue — priority and fairness aren't first-class concepts.

5. Our Project — Brief
FairQueue is a background job scheduler that executes jobs by priority while mathematically guaranteeing that no job waits longer than a bounded maximum time, using an aging algorithm borrowed from operating-systems process scheduling. Failed jobs are retried with exponential backoff, and jobs that fail repeatedly are moved to a dead-letter queue for manual inspection instead of retrying forever. The result: high-priority work gets done fast, but nothing is ever silently forgotten.

6. How This Differs From ScheduleIQ
ScheduleIQ decides who works which shift, solving a constraint-satisfaction problem over people and time slots using an external solver. FairQueue decides in what order background computational jobs run, solving a queueing-and-fairness problem using a scheduling algorithm you design and implement yourself — no external solver involved. ScheduleIQ's core challenge is finding one optimal assignment among trillions of possibilities; FairQueue's core challenge is guaranteeing fairness and reliability under continuous, unpredictable load over time. The underlying computer science — constraint programming versus real-time scheduling theory — is genuinely different, so the two projects don't overlap in what they prove about your skills.

7. How We Overcome Existing Solutions
Unlike Quartz/Spring Batch, FairQueue enforces a provable maximum wait time for every priority level through an aging mechanism — a low-priority job's effective priority increases the longer it waits, until it's guaranteed to run. Unlike Celery/SQS, priority and fairness are core to the scheduling algorithm, not an afterthought bolted onto a plain FIFO queue. Unlike basic queues, every failure path (retry, backoff, permanent failure) is handled explicitly and observably, not left to fail silently.

8. Tech Stack
Backend: Java, Spring Boot
Concurrency: Java PriorityBlockingQueue / custom priority queue with aging logic, ExecutorService thread pools
Persistence: PostgreSQL (job state, retry counts, dead-letter records)
Optional: Redis for distributed job locking if scaled across instances
Monitoring: Simple dashboard (Spring Boot Actuator or a lightweight React view) showing live queue state
9. Five Differentiating Features
Starvation-Free Priority Scheduling — an aging algorithm guarantees every job runs within a bounded maximum wait time, regardless of how many higher-priority jobs keep arriving.
Exponential Backoff with Jitter — failed jobs retry with increasing delay and randomized jitter, preventing retry storms from overwhelming downstream systems.
Dead-Letter Queue with Inspection API — jobs that fail beyond a retry limit are quarantined for manual review instead of retrying forever or silently disappearing.

1. Project Title
ForesightCache — Access-Pattern-Aware Predictive Cache

2. The Problem
Every high-traffic application relies on caching to stay fast — but standard caches are purely reactive: they only cache something after it's been requested once, meaning the very first request (and every "cold" request after eviction) is always slow. At scale, this "cold miss" problem costs real money and real latency — e-commerce platforms lose measurable revenue to slow page loads, and every extra millisecond of database load during peak traffic compounds. Most caching solutions accept this as an unavoidable cost of caching; very few attempt to reduce it by anticipating demand before it happens.

3. Existing Solutions
In-memory caches (Java HashMap/Guava Cache, Caffeine).
Distributed caches (Redis, Memcached).
CDN edge caching (Cloudflare, Akamai — for static content).
4. How They Win and How They Lose
Where they win: Caffeine/Guava are extremely fast and simple for single-instance apps. Redis/Memcached scale across distributed systems and support complex data structures. CDNs are excellent for static, rarely-changing content.

Where they lose: All of them are fundamentally reactive — they cache only after a request occurs. None of the mainstream open caching tools analyze historical access patterns to proactively warm the cache with data that's about to be needed. This means every cache, no matter how well-tuned its eviction policy, still suffers from repeated cold-start penalties on predictable, recurring access patterns (e.g., a product page that always spikes at 9 AM).

5. Our Project — Brief
ForesightCache is a caching layer that does everything a standard LRU/LFU cache does, plus one addition that most caches lack: it tracks historical access patterns by key and time, detects recurring demand spikes (e.g., "this key is heavily requested every day between 9–10 AM"), and proactively pre-loads that data into cache before the spike happens — turning a chunk of predictable cold misses into warm hits.

6. How This Differs From ScheduleIQ
ScheduleIQ optimizes a one-time (or periodic) resource assignment problem using a constraint solver, producing a single "best" schedule for a fixed planning window. ForesightCache is a continuously running, real-time system that observes a live stream of access events and adapts its behavior on an ongoing basis — there's no fixed planning window or single optimal answer, only continuous pattern detection and adaptation. The engineering discipline is different too: ScheduleIQ is about modeling constraints for a solver; ForesightCache is about systems performance, memory management, and time-series pattern recognition — a distinct, infrastructure-level skill set.

7. How We Overcome Existing Solutions
Unlike Caffeine/Guava/Redis, which only react to requests as they happen, ForesightCache adds a lightweight pattern-tracking layer that studies when and how often each key is accessed, then proactively warms the cache ahead of predicted demand — reducing cold misses that standard caches simply accept as unavoidable. It's not a replacement for Redis-style infrastructure; it's a genuinely novel layer most existing tools don't attempt, giving you a real point of technical differentiation to discuss.
